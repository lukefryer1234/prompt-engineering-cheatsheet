# 🤝 Contributing to Prompt Engineering Cheat Sheet 2026

First off, thank you for considering contributing! This cheat sheet is a community resource, and your input makes it better for everyone.

## 📋 How to Contribute

### 🐛 Report Issues

- Found outdated information? [Open an issue](../../issues/new)
- Spotted a typo or broken link? [Open an issue](../../issues/new)
- Have a suggestion for a new section? [Open an issue](../../issues/new)

### 📝 Submit Changes

1. **Fork** this repository
2. **Create a branch** for your changes:
   ```bash
   git checkout -b feature/add-new-technique
   ```
3. **Make your changes** following the style guidelines below
4. **Commit** with a clear message:
   ```bash
   git commit -m "Add: Reflexion technique to Core Techniques section"
   ```
5. **Push** and open a **Pull Request**

### 💡 What We're Looking For

- **New prompt techniques** with clear explanations and examples
- **Model-specific tips** based on real-world experience
- **Prompt templates** for common use cases
- **Anti-patterns** you've encountered
- **Cost optimization** strategies
- **Security** best practices
- **Corrections** to outdated information (models, pricing, capabilities)

## 📏 Style Guidelines

### Writing

- Use **clear, concise language** — this is a cheat sheet, not a textbook
- Include **practical examples** with every technique
- Use **tables** for comparisons and quick reference
- Use **emojis** sparingly for visual scanning (match existing style)
- Keep code examples **self-contained** and copy-paste ready

### Formatting

- Use `##` for main sections, `###` for subsections
- Wrap code in fenced code blocks with language identifiers
- Use tables for structured comparisons
- Link to external resources using `[text](url)` format

### File Structure

```
README.md           ← Main cheat sheet
examples/
  ├── code-generation.md
  ├── content-writing.md
  ├── data-analysis.md
  └── your-new-category.md   ← Add new example files here
```

### Commit Messages

Use the format: `Type: Description`

- `Add:` — New content or sections
- `Fix:` — Corrections or bug fixes
- `Update:` — Changes to existing content
- `Remove:` — Deleted content
- `Docs:` — Documentation-only changes

## 🏷️ Pull Request Guidelines

- **One topic per PR** — Keep changes focused
- **Include context** — Explain *why* the change is valuable
- **Test your markdown** — Preview renders correctly on GitHub
- **Update the Table of Contents** if adding new sections

## 📜 Code of Conduct

- Be respectful and constructive
- Focus on facts and practical experience
- Cite sources when referencing research or benchmarks
- No promotional content (except relevant, genuinely useful tools)
